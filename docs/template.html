webstack

A project to prototype web stacks source code

Each stack serves a healthcheck-style HTTP API endpoint that executes a SELECT 1 to a SQL database and responds with JSON {"status":"ok"}. Each stack uses a lightweight router, database connection pool, and SQL database driver (no ORM). Example: 

      
require "connection_pool"
require "pg"
require "sinatra"

pool = ConnectionPool.new(size: 5, timeout: 5) {
  PG.connect(ENV.fetch("DATABASE_URL"))
}

get "/" do
  pool.with { |db| db.exec("SELECT 1") }
  content_type :json
  {status: "ok"}.to_json
end

heroku

From 2009-2022, my stack most often included Heroku + Postgres:

  • Version code in a GitHub repo
  • Open GitHub pull requests w/ CI
  • Merge into main branch to auto-deploy to staging environment
  • Promote staging manually to production environment in a pipeline
  • Configure production database with a follower

Cons:

  • Heroku lacks overdue features.
  • It's GitHub integration broke April 2022 in incident 2413 and took over a month to restore.

fly

Pros:

  • I was able to get excellent performance and reliability from Fly.io.
  • The CLI is fantastic and seems to contain all info for the app.
  • It seems like Fly can understand a Procfile, which is helpful for not needing to brush up on Dockerfile layouts, which I'm rusty on.

Cons:

  • To deploy from my laptop (Apple M1 chip), I had to install Docker (~1.5GB). Without it, I observed errors. It seems like Docker should not have been required because the flyctl launch command's --remote-only flag is true by default.
  • To automatically deploy my main branch, I had to create a GitHub Actions workflow. Not really a big deal. Pretty easy to set up but not as "magical" as Railway's deploys.
  • The web UI is not as aesthetic as Railway, Heroku, or Render. There is more info in the CLI that could be in the UI.

railway

Pros:

  • The web UI is absolutely gorgeous.
  • Web services and Postgres are easy to set up and connect to each other. The UI provides a great visualization of their relationship.
  • Nice in-browser SQL editor / console.
  • Philosophically aligned with my general mentality of wanting to not think in containers.
  • They've been moving to a new packaging system they created called Nixpacks. They've rapidly added support for languages and builds seem pretty fast.

Cons:

  • SSL certs at set with a short expiration, which makes me nervous when reviewing my Checkly dashboard.
  • When testing Deno on Railway, the Postgres database driver threw `invalid peer certificate UnknownIssuer` errors and defaulted to a non-encrypted connection.

render

Render supports some key features that Heroku does not:

  • HTTP/2 support
  • IP access control on Postgres databases
  • DDoS protection
  • Health checks

Cons:

  • Render's free tier is not as generous as Fly's or Railway's. I've therefore kept fewer Render apps running and explored it less than Railway.
  • The default Ruby version in their environment is oddly 2.6.8. The 2.6.x series was superceded by the 2.7.x series on Christmas 2019 and by the 3.x series on Chrismas 2020.
  • Postgres databases are backed up daily, not continuously.
  • HA Postgres databases are not available.
